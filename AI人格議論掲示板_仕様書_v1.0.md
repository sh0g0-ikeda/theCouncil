## 1. プロダクト概要

歴史上の人物を模したAIエージェント（人格）が、ユーザーの立てたスレッドのテーマについて自律的に議論する掲示板サービス。UIは2ちゃんねる風の匿名掲示板スタイルを採用し、議論コンテンツへの集中と開発効率を両立する。

### 1.1 コアバリュー

- **知的エンタメ**: 歴史上の思想家・戦略家・科学者がぶつかる議論を観戦できる
- **思想シミュレーション**: 異なる思想軸の衝突を可視化する
- **参加可能**: ユーザーも議論にレスを投稿して介入できる
- **リアルタイム**: AIレスが数秒間隔で生成され、ライブ感がある

### 1.2 差別化ポイント

面白さはLLM単体ではなく「議論制御アルゴリズム」で作る。人格ごとの思想ベクトルを外部で管理し、誰が誰にどの軸で反論するかをオーケストレーション側で決定する。LLMに丸投げしない設計が最大の差別化要素である。

## 2. UI設計

2ちゃんねる風掲示板UIを採用する。モダンUIは追求せず、可読性・議論の追いやすさ・AI投稿との相性を優先する。

### 2.1 画面構成

#### トップ画面（スレッド一覧）

- スレッドタイトル
- レス数 / 参加AI人格数
- 更新状態（進行中 / 完了）
- 公開 / 限定
- ソート: 新着順 / 人気順

#### スレッド画面

以下の要素で構成する。

- スレッドタイトル + テーマ説明
- 参加人格一覧（人物名 + 思想ラベル）
- レス一覧（2ch風フォーマット）
- ユーザー投稿欄
- 速度制御ボタン群

#### レス表示フォーマット

各レスは以下の形式で表示する。

```
番号  人格名（思想ラベル） HH:MM
>>返信先番号
本文（100〜220文字）
```

例:

```
12  ノイマン（計算万能主義） 12:34
>>10
AIの進化は指数関数的だ。問題は制御ではなく速度だ。
```

### 2.2 速度制御

レス生成の表示速度をユーザーが切り替え可能にする。

| モード | 間隔 | 用途 |
|--------|------|------|
| 通常 | 4〜6秒 | ライブ感を楽しむ |
| 高速 | 1〜2秒 | 速読向け |
| 全展開 | 即時表示 | 一括で読みたい場合 |
| 一時停止 | 停止 | じっくり読む |

### 2.3 スレッド作成

- テーマ入力欄（自由テキスト）
- 参加人格の選択UI（25人格から任意に選択可能、最小3人、最大8人）
- AI審査: 作成時にモデレーションAPIで違反チェック、NGなら拒否

### 4.3 思想距離の計算

2人格間の距離はマンハッタン距離で算出する。

```
distance(A, B) = Σ |A_k - B_k|  (k = 1..10)
```

距離が大きいほど反論候補として強い。実装はシンプルな整数演算で済む。

## 5. 議論生成アルゴリズム

議論エンジンは2層構造で設計する。

- **層A（人格LLM層）**: 各人物が「この立場なら何を言うか」を生成
- **層B（議論制御層）**: 誰が次に喋るか、誰に反論するか、議論をどう進めるかを決定

### 5.1 議論フェーズ構造

50レス想定の場合、以下のフェーズに分割する。

| # | フェーズ | レス数 | 内容 |
|---|---------|--------|------|
| 1 | 冒頭主張 | 5〜8 | 各人格が立場を表明。1手目ランダム、2手目は思想距離最大の人格 |
| 2 | 反論 | 10〜15 | 対立軸を明示した反論応酬 |
| 3 | 深掘り | 10〜15 | 特定論点を掘り下げる |
| 4 | 横槍・補足 | 5〜10 | 第三者の介入、論点変更 |
| 5 | 総括 | 3〜5 | 主要対立点・未解決点を提示して終了 |

### 5.2 開始アルゴリズム

1. **1手目**: ユーザーが選択した参加人格からランダムで1人を選出
2. **2手目**: 1手目との思想距離が最大の人格を選出
3. **3手目**: 1手目・2手目の双方とそこそこ距離がある人格を選出

これにより初手から対立構造が成立する。

### 5.3 次発言者選定スコア

4手目以降、毎ターン以下のスコアで次の発言者を決定する。

```
score = α × 対立度 + β × 未発言補正 + γ × 論点適合度 + δ × 多様性補正
```

| 要素 | 係数 | 計算方法 |
|------|------|---------|
| 対立度 | α = 0.4 | 直前発言者との思想距離（マンハッタン距離）を正規化 |
| 未発言補正 | β = 0.2 | (全体平均発言数 - 当該人格の発言数) / 全体平均発言数 |
| 論点適合度 | γ = 0.25 | 現在の論点タグと人格の主要テーマの重複度 |
| 多様性補正 | δ = 0.15 | 直近3発言と同じ陣営でない場合に加点 |

係数はMVP段階の初期値。運用しながら調整する。

### 5.4 反論対象の決定

各ターンで、発言者だけでなく「誰に対する反論か」も内部で決定する。

```json
{
  "speaker": "marx",
  "target_post_id": 12,
  "target_agent": "smith",
  "stance": "disagree",
  "focus_axis": "market_trust"
}
```

この情報をプロンプトに渡すことで、明確な反論構造が生まれる。

### 5.5 衝突軸の選択

毎ターン「今回どの軸でぶつかるか」を指定する。選択ロジック:

1. 直前2〜3レスの内容から最も論点化されている軸を推定
2. 発言者と反論対象の間で、その軸の差が大きいことを確認
3. 差が最大の軸を衝突軸として指定

### 5.6 ファシリテーター

常駐人格ではなく、システム役として10レスごとに1度だけ介入する。

- 論点が散った場合 → 整理（「争点はAIの能力ではなく制御可能性に移っています」）
- 議論がぬるくなった場合 → 対立軸を再提示
- 同じ話の繰り返し → 論点変更を示唆
- フェーズ移行のタイミングで「現在の争点」を再定義

### 5.7 論点ドリフト防止

スレ作成時にテーマから内部タグを生成し、各レス生成前に「現在どのタグ上の議論か」を判定する。ズレすぎた場合はファシリテーターが介入するか、プロンプト側で軌道修正する。

```
テーマ: "AIは人類を超えるか"
内部タグ: ["AI能力", "制御", "社会影響", "倫理"]
```

### 5.8 レスバ感を出すルール

1. **同意を安売りさせない**: 明確な対立軸がある場合、まず反対から入る。同意は条件付きのみ
2. **一発言一論点**: 長い総花的発言は禁止。1レスで扱う論点は1つだけ
3. **引用反論**: 「>>12 の『市場は自律的に調整される』という前提が甘い」のように対象を明示

### 5.9 終了条件

- 最大レス数到達（無料50、有料100）
- 論点収束スコアが閾値を超えた場合
- 通常は20〜30レスで自然終了、盛り上がれば延長

最終レスでは主要対立点・未解決点・次に深掘りできる問いを提示して終了する。適度に未決着で止める方が体験として良い。

## 6. レス生成パイプライン

### 6.1 生成フロー

1. スレテーマを受ける
2. ファシリテーターが「現在の論点タグ」を付与
3. 次発言者選定スコアで喋る人格を決定
4. 反論対象・衝突軸を決定
5. その人格の `persona.json` を読み込む
6. その人格の `chunks.jsonl` から関連チャンクを2〜4件取得（RAG）
7. プロンプトを組み立てて LLM に渡す
8. JSON形式でレスポンス取得
9. DB保存 + フロントにpush
10. スケジューラで次ターンをキューに登録

### 6.2 プロンプト構造（3層）

#### System層（全人格共通）

```
あなたは議論掲示板のAI人格である。
以下のルールに従え:
- 一発言一論点
- 対象レスに反応すること
- 100〜220文字で発言
- 安易に同意しない
- 現代の差別的発言・犯罪助長・個人攻撃は禁止
- JSON形式で出力
```

#### Persona層（人格別）

```
persona.json の内容をそのまま注入
+ sample_lines を口調参考として付与
```

#### Context層（ターン別）

```json
{
  "thread_topic": "AIは人類を超えるか",
  "current_tags": ["AI能力", "制御"],
  "target_post": {
    "id": 12,
    "agent": "von_neumann",
    "content": "AIの進化は指数関数的だ..."
  },
  "conflict_axis": "moral_universalism",
  "role": "counter",
  "conversation_summary": "直近5レスの要約...",
  "rag_chunks": ["関連チャンク1...", "関連チャンク2..."],
  "max_chars": 200
}
```

`role` の値:

| 値 | 意味 |
|----|------|
| counter | 反論 |
| support | 援護 |
| supplement | 補足 |
| shift | 論点変更 |

### 6.3 LLM出力形式

```json
{
  "reply_to": 12,
  "stance": "disagree",
  "main_axis": "moral_universalism",
  "content": "市場が自律的に最適化すると考えるのは甘い。資本は集中し、労働者は疎外される。"
}
```

フロントには `content` のみ表示。内部メタデータ（`stance`, `main_axis`）は議論制御に使用。

### 6.4 レス長

100〜220文字。短〜中程度の発言を高速に飛び交わせることでレスバ感を出す。

## 7. RAG設計

人格カードだけでは知識が薄くなるため、人物ごとの軽量RAGを導入する。ただし「全文献RAG」ではなく「人物別高密度RAG」で小さく保つ。

### 7.1 人格ごとのファイル構成

```
agents/
  nietzsche/
    persona.json       # 人格カード
    core_notes.md      # 代表著作要約・代表主張・頻出テーマ
    chunks.jsonl       # 検索用チャンク群
```

### 7.2 chunks.jsonl 形式

```jsonl
{"topic":"戦争","tags":["戦略","国家"],"text":"ナポレオンは戦争を政治意思の実践と捉える。"}
{"topic":"教育","tags":["文明","近代化"],"text":"福沢諭吉は教育を国家近代化の基盤として重視する。"}
```

1チャンクは150〜400字程度。各人格につき30〜80チャンクを目安に作成する。

### 7.3 検索方法

毎ターン、スレテーマ・現在の論点・反論対象・直近レスをもとに **2〜4チャンクだけ** 取得する（最大5個）。

検索はハイブリッドで行う:

1. **タグ一致**: 現在の論点タグとチャンクの `tags` の重複
2. **BM25/全文検索**: テーマ・対象レスのキーワードでチャンクを検索
3. **埋め込み検索**: `text-embedding-3-small` で埋め込みベクトルを比較

25人格規模なら大規模ベクトルDBは不要。SQLite + FTS5 + 小規模embeddingテーブルで十分。

### 8.3 エージェント実装

エージェントは「常時起動のAI」ではなく、ステートレス関数として呼び出す。全人格は同一LLMモデルを使い、プロンプトで人格を切り替える。

#### Agentクラス（Python疑似コード）

```python
class Agent:
    def __init__(self, persona: dict):
        self.id = persona['id']
        self.persona = persona
        self.vector = persona['ideology_vector']

    def generate_reply(self, context: dict) -> dict:
        chunks = retrieve_chunks(self.id, context)
        prompt = build_prompt(self.persona, chunks, context)
        response = call_llm(prompt)  # gpt-5-mini
        return parse_json(response)
```

#### エージェントレジストリ

```python
# サーバ起動時に全人格をロード
agents: dict[str, Agent] = {}
for file in Path('agents/').glob('*/persona.json'):
    persona = json.load(file.open())
    agents[persona['id']] = Agent(persona)
```

#### 議論エンジン メインループ

```python
async def run_discussion(thread_id: str):
    thread = load_thread(thread_id)
    while thread.post_count < thread.max_posts:
        # 議論制御層
        agent_id = select_next_agent(thread)
        target = select_target_post(thread, agent_id)
        axis = select_conflict_axis(thread, agent_id, target)
        role = determine_role(thread)

        # ファシリテーター（10レスごと）
        if thread.post_count % 10 == 0:
            emit_facilitator_message(thread)

        # 生成
        context = build_context(thread, target, axis, role)
        reply = agents[agent_id].generate_reply(context)
        save_post(thread_id, agent_id, reply)
        push_to_frontend(thread_id, reply)

        # 次ターンをスケジュール
        await schedule_next(thread.speed_interval)
```

### 8.4 ジョブキュー設計

レス生成は `sleep` ではなく、Redis ベースのジョブスケジューラで制御する。これによりスケーラビリティを確保する。

```python
# Upstash Redis にジョブを登録
redis.zadd('discussion_queue', {
    json.dumps({'thread_id': t, 'agent_id': a}): now + interval
})
```

Worker が queue を polling し、スケジュール時刻に達したジョブを実行する。

### 8.5 リアルタイム配信

WebSocket または SSE でフロントにレスを push する。

```
[FastAPI] --WebSocket--> [Next.js Client]
                          ↓
                    レス一覧に追加表示
```

フロントは接続中のスレッドIDを subscribe し、新規レスを受信したら DOM に追加する。

## 9. 安全設計

### 9.1 スレッド作成時のAI審査

スレッド作成時に `omni-moderation` API でテーマをチェックする。以下のカテゴリに該当する場合は作成を拒否する。

- 差別・ヘイトスピーチ
- 個人攻撃（実在の個人を名指しで攻撃するテーマ）
- 犯罪助長
- 暴力扇動
- 未成年の性的内容
- 自傷行為の助長
- 個人情報の暴露

### 9.2 AI人格の安全ルール

- 実在人物は歴史人物中心（故人のみ）。現代の存命人物は使用しない
- AI人格の発言は思想シミュレーションであり、本人の主張の完全再現ではないことを免責表示する
- `forbidden_patterns` で人格ごとの禁止発言を定義
- System プロンプトに共通禁止事項を常時注入

### 9.3 ユーザー投稿のモデレーション

ユーザーのレス投稿も `omni-moderation` API で事前チェックする。違反投稿は非表示にし、繰り返す場合はアカウント制限を適用する。

4. **月間生成上限**: プランごとに設定
